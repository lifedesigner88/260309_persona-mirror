## 3. Fix: Admin Seed Crash on Restart (Duplicate Email)

### What Changed
- `sync_admin_seed` in `auth/service.py`: existence check changed from `WHERE user_id = ?` to `WHERE user_id = ? OR email = ?`

### Why It Mattered
When `ADMIN_SEED_USER_ID` env var changed between deploys (e.g. `"admin"` â†’ `"ADM001"`), the old row was not found by `user_id`, triggering an unconditional INSERT that violated the unique constraint on `email`. Backend crash-looped on every restart against Supabase. Checking by email as well makes the seed idempotent regardless of `user_id` changes.

- `capture/schemas.py`: `owner_user_id: str â†’ int`
- `seed.py`: removed `DEMO_USER_ID`; finds user by email; `Persona(owner_user_id=...)` â†’ `Persona(user_id=user.user_id, ...)`
- `.env.example`: removed `ADMIN_SEED_USER_ID`

**Frontend (`apps/frontend/`):**
- `auth/types.ts`: `user_id: number`, removed `generatedUserId`
- `auth/api.ts`: removed `user_id` from signup body; returns `signupEmail` only on success
- `auth/pages/signup-page.tsx`: removed 6-char ID card (generate/copy/shuffle UI), Step 2 ID display box, Step 3 ID confirmation screen

Replacing it with a DB auto-increment integer removes the ID card step from signup entirely â€” users sign up with email + PIN only. `user_id` is now an internal integer returned after signup, not a user-visible concept. The `users` table column rename from `id â†’ user_id` (as sole PK) also simplifies all FK references.
