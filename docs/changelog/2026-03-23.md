## 1. Supabase as Managed DB

### What Changed
- `apps/backend/.env` + `apps/ai-worker/.env`: `DATABASE_URL` switched to Supabase connection string (ap-northeast-2 pooler, port 6543)
- `docker-compose.yml`: removed `db` (Postgres) and `redis` services; removed hardcoded `DATABASE_URL` overrides from `backend` and `ai-worker` environment blocks
- `app/common/db.py`: added `prepare_threshold=None` for Supabase PgBouncer (transaction pooler does not support prepared statements)

### Why It Mattered
Local Postgres added friction for deployment — every environment needed its own volume and init. Supabase free tier handles the DB externally; `docker compose up` now starts 3 services instead of 5. pgvector is available on Supabase when needed for persona matching (M3).

## 2. Production Docker Images + GHCR CI

### What Changed
- `apps/frontend/Dockerfile`: converted to multi-stage build — `node:24.11.0-alpine` builder runs `pnpm build`, output copied into `nginx:1.27-alpine`; `VITE_API_BASE_URL` passed as build arg
- `apps/frontend/nginx.conf`: added `try_files $uri $uri/ /index.html` for SPA routing
- `.github/workflows/build-images.yml`: on push to `main`, builds and pushes `persona-mirror-frontend`, `persona-mirror-backend`, `persona-mirror-ai-worker` to GHCR with layer caching

### Why It Mattered
The old Dockerfile ran `pnpm dev` — a dev server not suitable for production. Multi-stage build produces a lightweight nginx image with baked-in static assets. `VITE_API_BASE_URL` is baked at build time so the browser knows where to reach the backend (`https://asm17.huposit.kr/api`).

## 3. Fix: Admin Seed Crash on Restart (Duplicate Email)

### What Changed
- `sync_admin_seed` in `auth/service.py`: existence check changed from `WHERE user_id = ?` to `WHERE user_id = ? OR email = ?`

### Why It Mattered
When `ADMIN_SEED_USER_ID` env var changed between deploys (e.g. `"admin"` → `"ADM001"`), the old row was not found by `user_id`, triggering an unconditional INSERT that violated the unique constraint on `email`. Backend crash-looped on every restart against Supabase. Checking by email as well makes the seed idempotent regardless of `user_id` changes.
